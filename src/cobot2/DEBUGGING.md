## 2. 시스템 파이프라인 (최종 아키텍처)

최종 마스터 서비스 노드(`realtime_3d_mapper_multi_10_01.py`)의 실시간 처리 흐름:

### 1단계: ROS 2 Trigger 서비스 개시
- 메인 오케스트레이터가 로봇(M0609)을 5개 작업대(Workstation)에 순차 배치
- 로봇 정지 시 서비스 클라이언트가 `/vision_inspect` (std_srvs/srv/Trigger)를 비동기 호출

### 2단계: YOLO 비전 스캔 및 공정 최적화
- 2D 컬러 이미지 수신 즉시 YOLOv11x 모델을 구동하여 나사 존재 여부 판정
- **나사 미발견 시** `SKIPPED` 리턴 → 저장/통신 연산을 원천 차단하여 대역폭 및 Storage 낭비 방지

### 3단계: 핀홀 카메라 기반 3D 물리 좌표 복원
- CameraInfo의 렌즈 파라미터(fx, fy, cx, cy)와 Aligned Depth 데이터를 결합
- 역투영 공식 `x = (u - cx) × z / fx` 을 직접 연산하여 3D 카메라 좌표 복원
- ROS 2 TF2로부터 취득한 4×4 변환 행렬을 적용하여 로봇 base_link 좌표로 정밀 전환

### 4단계: 동적 도넛 알고리즘 — 나사 단차(mm) 품질 검사

**4-1. 전역 작업대 평면 정의**
- PointCloud 기반 RANSAC 알고리즘(`fit_plane_ransac`)으로 작업대 표면의 평면 방정식(법선 벡터, 평면 위 한 점)을 도출하여 '바닥 기준면' 설정

**4-2. 나사 머리 영역 정밀 추출**
- `get_robust_screw_center` 함수에서 HSV 색상 필터링 → 명도(V) 90 이하의 검은색 영역만 추출
- 바닥 평면 기준 5mm ≤ 높이 ≤ 30mm인 점들만 유효한 나사 후보로 선별 (노이즈·그림자 제외)

**4-3. 최종 돌출 높이(단차) 산출**
- 유효 나사 점군 중 가장 밀집된 메인 컴포넌트를 선택
- 바닥 평면 수직 방향 최댓값(`np.max(selected_heights)`)을 나사 최종 돌출 높이로 확정

**4-4. 합격/불량 판정**
```python
status_bool = bool(delta_z_mm <= 10.0)
# 정상: 돌출 높이 ≤ 10.0mm
# 불량: 돌출 높이 > 10.0mm → defect_type = 'height_defect'
```

### 5단계: 클라우드 동기화 및 디지털 트윈 저장

원거리 배경 노이즈를 0.25m~1.0m 이내로 사전 커팅하여 용량 최소화 후, 5개 모듈화 함수로 Firebase에 분산 저장:

| 모듈 | 경로 | 역할 |
|------|------|------|
| `_update_live_scan` | `live_scan/workstations/{id}` | 실시간 모니터링용 — 작업대 상태, 3D 배경 URL, 나사별 합불·좌표·로봇팔 ZYZ 회전각 |
| `_update_inspection_records` | `inspection_sessions/{session_id}/...` | 영구 아카이브 — metadata, summary, markers, TF 변환 행렬 스냅샷 |
| `_update_twin_states` | `twin_state/`, `sites/`, `robots/` | 디지털 트윈 상태 — 누적 통계, 로봇 모드를 `inspection_completed`로 전환 |
| `_update_database_indexes` | `indexes/` | 빠른 조회용 인덱스 — 날짜별·작업대별 매핑, 불량 추적 인덱스 |
| `_update_events` | `events/{event_id}` | 시스템 로그 — 불량 0건이면 `info`, 1건 이상이면 `warning`으로 격상 |

## 4. 프론트엔드 웹 관제 대시보드 고도화 이력

### 웹 데이터 흐름 아키텍처

```
Firebase Realtime DB → live_scan/workstations 실시간 트리거
       ↓
동적 스크립트 인젝션 → background_url 감지 시 3D 점구름(.js) DOM 비동기 로드
       ↓
데이터 캘리브레이션 → 'm' 단위 데이터 유입 시 'mm'으로 실시간 스케일 변환
       ↓
Plotly 3D 동적 렌더링 → scatter3d 하드웨어 가속 + 드래그/오비트 카메라 앵글
       ↓
상태 하향식 제어(HRI) → 작업자 '해결 완료' 클릭 시 DB 역업데이트 → 코봇 인터록 해제
```

### viewer_01 → 비동기 렌더링 체계 구축
- **문제:** 나사 상태 변경 시마다 대용량 배경 스크립트를 매번 재다운로드 → 브라우저 Freeze
- **해결:** `bgUrl !== currentBgUrl` 조건 비교 레이어 설계. 배경 URL 변경 시에만 `<script>` 재생성, 마커만 변경된 경우 캐싱된 `window.latestBackground` 사용

### viewer_02 → 3D 마우스 인터랙션 마비 해결
- **문제:** Plotly 기본 설정으로 마우스 드래그 시 회전이 아닌 Pan 이동만 작동 → 결함 다각도 검사 불가
- **해결:** `dragmode: 'orbit'`, `scrollZoom: true` 명시적 활성화로 완벽한 자유 3D 회전 제어 달성

### viewer_03 → 3D 차원 왜곡 및 초기 앵글 최적화
- **문제:** Plotly 엔진이 3D 데이터를 2D 평면으로 오인 렌더링 / 초기 정면 축 정렬로 입체감 부재
- **해결:** `type: 'scatter3d'` 하드코딩으로 차원 왜곡 차단 + `scene.camera.eye: {x:1.5, y:1.5, z:1.2}` 탑다운 뷰포트 각도 주입

### viewer_04~05 → 공장 현장 시인성 개선
- **문제:** 촘촘한 3D 점구름 속에 나사 결함 마커와 텍스트가 파묻혀 현장 모니터에서 식별 곤란
- **해결:** 마커 크기 18, 텍스트 크기 24, Arial Black 폰트 고정, `textposition: 'top center'` 오프셋 재설정으로 시인성 획기적 개선

### viewer_06 → 멀티 워크스테이션 및 단위 호환성 통합 (최종)
- **문제:** ① 단일 `linestatus` 구조로는 5개 작업대 선택 관제 불가 ② 구형(m) / 신형(mm) 단위 데이터 혼재 → 맵 증발 및 스케일 비틀림
- **해결:** `live_scan/workstations` 경로 전환 + 동적 드롭다운 셀렉터(`<select>`) 탑재 → 오퍼레이터가 작업대 1~5를 자유 선택 관제. 배경 데이터 첫 번째 좌표 절대값 검사 → `< 10.0`이면 `.map(v => v * 1000.0)`으로 mm 자동 업스케일링
