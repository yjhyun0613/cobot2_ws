Overview

프로젝트 개요 (JoinGo)

제조 공정 내 다중 워크스테이션 상에서 조립되는 가공물들의 정밀한 형상 관리와 볼트 체결 불량 검출은 조립 품질 보증의 핵심적인 요소입니다.

본 프로젝트는 두산 협동로봇(M0609) 끝단에 장착된 Intel RealSense D435i 카메라를 활용해, 5개의 분산된 작업대(Workstation 1~5)를 순회하며 3차원 포인트 클라우드(Point Cloud) 배경 데이터와 고해상도 RGB-D 이미지를 동시 취득합니다.

취득한 2D 비주얼 박스 위치 정보에 뎁스 센서 기반의 공간적 깊이 지능(Spatial Intelligence)을 결합하여 가공물 표면 대비 볼트 체결부의 상대적 3D 돌출량(단차)을 mm 단위로 정교하게 연산하며, 오차를 클라우드로 실시간 중계하여 3D 디지털 트윈 맵으로 시각화합니다.

"YJH(연제현) 담당 영역: 비전 코어 & 3D 디지털 트윈 HRI"

협동 로봇이 각 작업대 이동을 완료하고 쏘는 신호를 받아 구동되는 ROS 2 AI 비전 서버 노드 전체 개발, 3D 기하복원 및 평평화 보정 알고리즘 수립, Firebase IoT 데이터 동기화 파이프라인, 그리고 작업자가 웹 상에서 다중 라인을 스위칭하고 불량을 마우스 클릭만으로 해제할 수 있는 Plotly.js 기반 3D 웹 관제 대시보드 일체를 설계하고 최적화했습니다.

Open3D 시각화 독점 스레드 락

발생 현상 (Symptom)

로봇 제어 PC 내부에서 Open3D GUI를 가동하는 동안 UI 렌더링 루프가 메인 ROS 스레드를 완전히 점유(Blocking)하여, 외부 관제 브라우저와의 3D 맵 공유가 원천 불가능한 상태 발생.

원인 분석 (Root Cause)

Open3D의 윈도우 그래픽 런타임은 싱글 스레드 중심 렌더링 메커니즘을 요구하며, 웹 브라우저에서 동적으로 연동하기에 무겁고 비표준적인 구조를 지니고 있었습니다.

해결 방안 (Solution)

Plotly.js 교체 및 독립 HTML 렌더러 구축: Open3D UI 종속성을 삭제하고, JSON 데이터를 비동기로 받아 3D 하드웨어 가속(Web GL)으로 렌더링하는 Plotly.js 프레임워크로 프론트엔드를 통째로 이전했습니다.

# 백엔드에서 Plotly에 로드될 수 있도록 좌표 객체화 후 파일 내보내기 bg_dict = { 'x': np.round(pts_flattened[:, 0], 4).tolist(), 'y': np.round(pts_flattened[:, 1], 4).tolist(), 'z': np.round(pts_flattened[:, 2], 4).tolist(), 'colors': colors_list } # bg_YYYYMMDD_HHMMSS.js 형태로 스토리지 저장 후 웹에서 dynamic script injection 로드

대용량 포인트클라우드 스트리밍 폭주

발생 현상 (Symptom)

30Hz로 밀려오는 카메라 포인트 클라우드를 실시간 데이터베이스에 그대로 업로드하다가 메모리 버퍼 오버플로우로 로봇 제어 시스템이 다운되고 클라우드 통신 트래픽 폭주.

원인 분석 (Root Cause)

초당 수백만 개의 3D 점 좌표를 끊임없이 원격 서버에 동기화하려고 한 설계적 문제 및 주변 공장 벽면 등의 불필요 노이즈 영역까지 모두 연산에 편입시켰기 때문입니다.

해결 방안 (Solution)

정지 스냅샷(Snapshot) 및 NumPy 복셀 슬라이싱: 지속적인 실시간 맵핑을 포기하고, 로봇이 목표지점 정지 완료 상태일 때 1회의 확정 프레임만 받아 RANSAC 피팅을 적용하는 "스냅샷 메커니즘"으로 선회했습니다. 또한 0.2m~1.0m 뎁스 윈도우 필터를 두어 허공의 배경 노이즈를 필터링했습니다.

# 거리 기반 필터링 및 다운샘플링 코드 distances = np.linalg.norm(pts, axis=1) mask = (distances >= 0.25) & (distances <= 1.0) # 유효 범위 설정 pts_filtered = pts[mask] # voxel downsample 생략 및 RANSAC 테이블 평평화 보정(Flattening) 적용

금속 표면 난반사 뎁스 유실 (NaN)

발생 현상 (Symptom)

나사 머리 표면 및 와셔부의 미세한 금속성 광택으로 인해 depth 정보가 일시적으로 잡히지 않고 NaN(도넛 현상) 처리되어, 볼트 높이가 비정상으로 나오거나 인덱스 오류로 정지되는 현상.

원인 분석 (Root Cause)

구조광 방식 D435i 카메라 특성상 은백색 금속 광택 면의 난반사 유입 시 적외선 위상 분석 축이 어긋나 깊이가 0 혹은 NaN 값으로 매핑되는 광학적 한계였습니다.

해결 방안 (Solution)

11x11 윈도우 스마트 주변 탐색 & Robust 3D Centering: 픽셀 중심점이 NaN일 경우 주변 11x11 픽셀 매트릭스를 회돌이하며 정상 거리값이 찍힌 점들의 미디언 필터 값을 취득했습니다. 또한 2D YOLO Box 크기에 맞춰 HSV 블랙 마스킹을 적용해 검은 영역의 connected components 분석만 추출해 강하게 매칭시켰습니다.

# fallback_median_pt 계산 로직 for v_idx in range(depth_patch.shape[0]): for u_idx in range(depth_patch.shape[1]): z_val = float(depth_patch[v_idx, u_idx]) / 1000.0 if 0.1 < z_val <= 2.0: valid_depths.append(z_val) # 중앙 미디언 실측값 추적 median_z = np.median(valid_depths)

Class-agnostic NMS 중복 감지

발생 현상 (Symptom)

카메라 앵글의 조명 차이 등으로 하나의 볼트에 여러 개의 바운딩 박스가 다중 중복 검출되어, 실제 4개의 나사가 놓인 공간을 5~6개의 검사 나사로 카운팅하는 오작동 발생.

원인 분석 (Root Cause)

YOLO AI 모델 추론 시 검출 바운딩 박스의 경계 임계치 융합(NMS)이 클래스별로만 독립적으로 작용하여 겹침 영역을 병합하지 못했기 때문입니다.

해결 방안 (Solution)

Class-agnostic NMS 도입: 나사 판정 클래스와 무관하게 두 Bounding Box의 겹치는 비율(IoU > 0.4 또는 IoMin > 0.6)을 선별 계산하여 신뢰도가 낮게 나온 다중 겹침 박스를 강제로 탈락시키는 필터를 수립했습니다.

# Class-agnostic NMS 필터링 예시 iou = inter_area / union_area min_area = min(area1, area2) iomin = inter_area / min_area if iou > 0.4 or iomin > 0.6: overlap = True # 신뢰도 높은 것만 살리고 제외

데이터 축 단위 파편화 (m vs mm)

발생 현상 (Symptom)

과거 빌드된 백엔드 로그(m 단위 기입)와 최신 빌드 백엔드(mm 단위 기입) 데이터가 DB에 혼재되어 있어, 웹 3D 화면 로딩 시 공간 축 스케일이 1000배 비틀어지거나 허공으로 튕기는 비주얼 크래시 발생.

원인 분석 (Root Cause)

서로 다른 ROS 2 노드 버전 간의 coordinate scaling 정책 불일치로 인해 저장 규격 파편화가 발생한 것이 원인이었습니다.

해결 방안 (Solution)

스케일 자동 캘리브레이션 필터 설계: 3D 배경 스크립트를 파싱하는 시점에 배경 x축 첫 번째 값의 크기를 읽어 `10.0` 미만인 경우(미터) 전체 배열 값을 자동으로 `* 1000.0` 하여 밀리미터 단위로 맞추는 지능형 호환 필터를 웹 클라이언트에 내재화했습니다.

// 지능형 단위 호환성 변환 레이어 (Web Front-end) if (bgDict) { let bgX = bgDict.x; if (bgX && bgX.length > 0 && Math.abs(bgX[0]) < 10.0) { // m 판정 bgX = bgX.map(v => v * 1000.0); bgY = bgY.map(v => v * 1000.0); bgZ = bgZ.map(v => v * 1000.0); } }
